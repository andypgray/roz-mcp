# Roslyn MCP Server Evaluation — nopCommerce

**Date:** 2026-03-22
**Evaluator:** Claude Opus 4.6 (1M context)
**Target:** nopCommerce open-source solution (37 projects, 3,489 files, C# 13 / .NET 9)
**Method:** Two passes — initial functional evaluation of all 22 tools, followed by a structured stress test designed to find bugs, performance limits, and UX friction.

---

## Overall Assessment

**Rating: 9/10** — production-ready for LLM use.

Across ~80 tool calls testing all 22 tools, there were zero crashes, zero timeouts, and zero garbled responses. The server handled extreme scale (4,478 references, 150-type hierarchies, 394-file renames) without issue. The design reflects deep understanding of how an LLM navigates code. The issues found are all fixable; none block productive use.

---

## Bugs

### 1. `remove_symbol` eats adjacent blank lines (MEDIUM)

Removing a symbol consumes the blank line before it. Re-inserting with `insert_before_symbol` doesn't restore it. A remove+insert round trip leaves the file with fewer blank lines than the original.

**Reproduction:**
```
# Original file has blank line between GetProductByIdAsync and GetProductsByIdsAsync.
# After remove_symbol + insert_before_symbol:

git diff:
-    Task<Product> GetProductByIdAsync(int productId);
-
+    Task<Product> GetProductByIdAsync(int productId);
     Task<IList<Product>> GetProductsByIdsAsync(...);
```

Confirmed in both evaluation passes. Two blank lines lost in a single cycle.

**Impact:** Formatting degrades across edit cycles. Diffs show spurious whitespace changes.

### 2. `rename_symbol` to same name reports false positive (MEDIUM)

Renaming `BaseEntity.Id` to `Id` reports "Changed 394 files" and triggers workspace reload. `git diff` shows zero actual changes.

**Reproduction:**
```
rename_symbol(filePath: "Libraries/Nop.Core/BaseEntity.cs", line: 11, column: 16, newName: "Id")
-> "Renamed 'Id' to 'Id'. Changed 394 file(s): [394 files listed]"
-> git diff --stat: (empty)
```

**Impact:** An LLM may believe it modified 394 files when nothing changed.

### 3. `find_overloads` doesn't recognize `this[]` for indexers (LOW)

Despite documentation listing `this[]` as a special name for indexers:

```
find_overloads(symbolName: "this[]", containingType: "CustomValues")
-> "No method found with name 'this[]'"
```

Workaround: `find_symbol(names: ["CustomValues"], depth: 1, memberKinds: ["Indexer"])` works.

### 4. `includeGenerated` flag has no visible effect on RateClient.cs (LOW)

Both `includeGenerated: false` and `true` return identical results for the auto-generated `RateClient.cs` (6,020 lines, 13 `#pragma warning disable` directives).

```
find_symbol(names: ["RateClient"], includeGenerated: false)  -> shows both partial locations
find_symbol(names: ["RateClient"], includeGenerated: true)   -> identical result
```

The file may not be classified as "generated" by the heuristic (no `<auto-generated>` comment or `[GeneratedCode]` attribute).

### 5. `remove_symbol` requires `filePath` even when symbolName is unambiguous (LOW)

```
remove_symbol(symbolName: "GetProductByIdAsync", containingType: "IProductService")
-> Error: missing required parameter 'filePath'
```

Other tools (`find_callers`, `find_references`, etc.) work fine with just `symbolName` + `containingType`.

## Design & UX Issues

### 6. `containingType` cannot disambiguate top-level types by namespace

When two types share a name (`Product` in Nop.Core vs Nop.Plugin.Misc.Zettle), the natural move is `containingType: "Nop.Core.Domain.Catalog"`. This is rejected — `containingType` only accepts type names. Top-level types have no containing type, so the only path is `filePath+line+column`.

**Suggestion:** Accept fully-qualified type names (e.g. `symbolName: "Nop.Core.Domain.Catalog.Product"`) or add a `namespace` parameter.

### 7. `find_symbol` default `Contains` match returns noisy partial hits

Searching for `"IProductService"` also matches `ImportProductImagesUsingServicesAsync`. Searching for `"ProductService"` returns 9 results including `CompareProductsService`, `CopyProductService`, etc.

**Suggestion:** Consider `StartsWith` as default, or rank exact/prefix matches first.

### 8. `go_to_definition` in doc comments does NOT reject — contradicts documentation

Documentation says "positions in comments and doc comments are rejected (not snapped to nearest symbol)." Reality: `go_to_definition` at line 8 col 5 (`/// <summary>` for `Id` property) resolves to `Id` on line 11. The tool snaps through the doc comment trivia to the owning symbol.

This behavior is actually **useful** — but the docs should be corrected.

### 9. `find_callers` on overloaded methods requires position disambiguation

`find_callers(symbolName: "GetAsync", containingType: "IStaticCacheManager")` with 4 overloads returns an ambiguity error listing all candidates. This is correct behavior — but means overloaded methods always need `filePath+line+column` for `find_callers`, while `find_overloads` works by name.

The error message is excellent (lists all candidates with qualified signatures and locations).

### 10. Enum members display as `[public static field]`

`CacheEventConsumer`'s nested `EntityEventType` enum shows members as `[public static field] EntityEventType Insert`. Technically correct (IL representation) but confusing — `[enum value]` would be clearer.

## What Works Excellently

### Name-based symbol targeting
`find_callers(symbolName: "GetProductByIdAsync", containingType: "IProductService")` just works — no file reading needed first. This eliminates the most common round-trip in code navigation and is the single biggest productivity win.

### Incremental diagnostics
`reset_baseline` → edit → `get_diagnostics(incremental: true)` is perfect. Remove a method → "334 new errors." Re-add it → "0 new errors." The diagnostic summary groups by error code and top affected files — ideal for triage.

### `find_callers` with `contextLines`
Each caller shows the enclosing method name plus surrounding code. Immediately useful without follow-up Reads. The output format (file → method → lines with context) is perfectly structured.

### `find_derived_types` tree rendering
150 `BaseEntity` subtypes rendered with `├─`/`└─` showing intermediate abstract classes (`BaseAttribute` → `AddressAttribute`, `CheckoutAttribute`, etc.). Beautiful and informative.

### Pagination hints
`(showing 5 of 245 total — excludeOverrides, or excludeTests)` tells both magnitude AND how to narrow. Actionable, not just informational.

### Auto-truncation with detail reduction
When output exceeds 80K chars, it strips source bodies and notes "Source bodies removed." Smart degradation, not failure.

### Batch `find_symbol`
`names: ["IProductService", "ICategoryService"]` in one call with `=== Search: "..." ===` headers. Per-name results with clear separation; misses reported individually.

### Disambiguation errors
Ambiguous `BaseEntity` (test code has a nested `CrudData<T>.BaseEntity`) → clear error listing both candidates with qualified names and locations.

### `add_usings` global using detection
`add_usings(usings: ["System.Linq"])` → "Already available via global using: System.Linq". Prevents redundant usings.

### `replace_content` CRLF normalization
Multi-line literal search works on Windows without line-ending concerns.

### Edit + immediate query consistency
After `replace_content`, `find_symbol` immediately finds new symbols. Zero lag.

### `(23 params)` summary
Complex signatures show parameter count instead of all types. `SearchProductsAsync(23 params)` is readable where the full signature would not be.

### `renameOverloads` flag
4 `GetAsync` overloads all renamed to `FetchAsync` across 63 files in one call. Clean and correct.

### Workspace info
37 projects with types, frameworks, nullable settings, file counts, and dependency graph. Right level of detail for orientation.

## Suggestions

### High Priority

| # | Suggestion | Rationale |
|---|---|---|
| 1 | Fix blank line preservation in `remove_symbol` | Formatting degrades across edit cycles |
| 2 | Detect no-op renames and skip | "Changed 394 files" when nothing changed is misleading |
| 3 | Accept qualified names or namespace filter for type disambiguation | Eliminates the most common extra round-trip |

### Medium Priority

| # | Suggestion | Rationale |
|---|---|---|
| 4 | Fix `this[]` in `find_overloads`/`find_symbol` | Documentation promises this works |
| 5 | Make `filePath` optional in `remove_symbol` when unambiguous | Consistency with read-only tools |
| 6 | Move total count to header in `find_callers`/`find_references` | "Callers of X (showing 5 of 245 total):" rather than at bottom |
| 7 | Increase `maxFiles` default from 20 to 30-50 | `*Controller.cs` globs hit 20 on first try |
| 8 | Correct docs: `go_to_definition` snaps through doc comments | Current behavior is useful but undocumented |

### Low Priority

| # | Suggestion | Rationale |
|---|---|---|
| 9 | Edit tool column hint on whitespace error | "Did you mean column 19 (`Name`)?" saves a Read |
| 10 | `get_workspace_info` compact mode for deps | 37-project dependency listing is token-heavy |
| 11 | `replace_content` report matched line numbers | Verify regex hit the right targets |
| 12 | `get_symbols_overview` return partial results at `maxFiles` limit | Partial > error for read-only exploration |
| 13 | Display `[enum value]` instead of `[public static field]` | Clarity |
| 14 | Consistent `(set maxMembers to see more)` hint across tools | `get_symbols_overview` has it, `find_symbol` doesn't |
| 15 | Document that nuclear renames may introduce errors | Roslyn limitation worth calling out |
| 16 | Consider `StartsWith` as default match mode | Reduces noise on targeted lookups |
